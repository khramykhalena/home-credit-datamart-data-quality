# Ключевые выводы по качеству данных

## application_train.csv
- Главная таблица витрины: `1 строка = 1 заявка`.
- Дубликатов по `SK_ID_CURR` нет.
- Класс целевой переменной несбалансирован: доля `TARGET = 1` заметно ниже доли `TARGET = 0`.
- В части колонок наблюдается высокий уровень пропусков, особенно в блоке жилищных признаков.
- Значение `DAYS_EMPLOYED = 365243` выглядит как техническая заглушка; для расчётов его нужно обрабатывать отдельно.

## bureau.csv
- На одного клиента приходится несколько записей о внешних кредитах; перед объединением требуется агрегация.
- Дубликатов по `SK_ID_BUREAU` нет.
- Пропуски особенно заметны в `AMT_ANNUITY`, `AMT_CREDIT_MAX_OVERDUE`, `DAYS_ENDDATE_FACT`, `AMT_CREDIT_SUM_LIMIT`.
- Есть строки с просрочкой и случаи, где `AMT_CREDIT_SUM_DEBT > AMT_CREDIT_SUM`; такие записи нужно интерпретировать осторожно и явно описать в правилах витрины.

## Практический вывод
Для BI и системной аналитики нельзя использовать сырые таблицы напрямую: сначала нужен слой data quality checks и агрегирующая витрина.


## previous_application.csv
- Один клиент может иметь несколько прошлых заявок; перед объединением с базовой таблицей нужен агрегирующий слой.
- Полезные для витрины признаки: число прошлых заявок, число одобрений и отказов, суммы прошлых заявок и кредитов, средняя глубина истории по `DAYS_DECISION`.
- В этой таблице особенно важно отслеживать статусы `Approved`, `Refused`, `Canceled`, `Unused offer`, потому что они напрямую помогают описать историю взаимодействия клиента с кредитным процессом.

## Практический вывод после добавления previous_application
Витрину уже можно усиливать не только внешней кредитной историей (`bureau`), но и собственной историей прошлых заявок клиента. Это делает проект заметно сильнее и для BI, и для системной аналитики.
